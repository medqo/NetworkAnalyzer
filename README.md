# ネットワーク診断アシスタント

Cisco Packet Tracerで構築したネットワークを対象に、CLIコマンドの実行結果を解析し、ネットワーク障害の原因を段階的に切り分けるStreamlitアプリケーションです。

OSI参照モデルに基づいて **L1 → L2 → L3** の順に診断を行い、障害を検出した場合は原因と復旧方法を提示します。

---

## 1. 概要

ネットワーク障害が発生した際、利用者が症状を選択すると、アプリケーションが確認対象の機器と実行すべきCisco CLIコマンドを順番に提示します。

利用者はPacket Tracer上で指定されたコマンドを実行し、その結果をアプリケーションに入力します。

入力されたコマンド出力を解析し、あらかじめ定義した正常なネットワーク構成と比較することで、設定の不足や異常を検出します。

### 診断の基本的な流れ

```text
症状を選択
    ↓
L1の診断
    ↓ 正常
L2の診断
    ↓ 正常
L3の診断
    ↓
診断完了
```

各診断項目で異常を検出した場合は、その時点で診断を終了し、以下を表示します。

- 障害内容
- 原因
- 復旧手順
- 復旧コマンド例

すべての診断項目が正常だった場合は、IPアドレスやDefault Gatewayなど、今回の自動診断対象外となるその他の原因候補を提示します。

---

## 2. 主な機能

### 症状に応じた診断

発生している症状を選択すると、その症状に対応した診断手順を開始します。

主な症状は以下のとおりです。

- 同じVLAN内の端末同士で通信できない
- 異なるVLAN間で通信できない
- Router-on-a-Stickが動作しない

### 段階的な障害切り分け

OSI参照モデルに基づいて、L1から順番に診断します。

前の診断項目が正常だった場合のみ次の項目へ進み、障害を検出した時点で原因を特定します。

### CLI出力の解析

以下のようなCisco CLIコマンドの出力を解析します。

```text
show interfaces status
show vlan
show interfaces trunk
show running-config
show ip interface brief
```

### 正常構成との比較

必要なVLAN、Trunkの許可VLAN、Subinterface、IEEE 802.1Q設定などを正常時の期待値として定義しています。

コマンド出力から必要な情報を抽出し、期待値と比較することで異常を判定します。

### 不足している設定の特定

単純に正常・異常を判定するだけでなく、不足しているVLAN番号なども取得します。

例えばVLAN30用のSubinterfaceが存在しない場合は、以下のように原因を特定できます。

```text
障害内容
Subinterface設定エラー

原因
VLAN [30] 用のSubinterfaceがありません
```

### 復旧方法の提示

検出した障害内容に応じて、復旧手順とCisco CLIコマンド例を提示します。

不足しているVLAN番号などの情報を利用することで、対象となる設定に応じた復旧コマンドを生成します。

## 3. 「異なるVLAN間で通信できない」の診断手順

この症状では、以下の5項目を順番に診断します。

| 順番 | レイヤー | 対象機器 | 確認コマンド | 確認内容 |
|---|---|---|---|---|
| 1 | L1 | SW1 | `show interfaces status` | Interfaceの接続状態 |
| 2 | L2 | SW1 | `show vlan` | 必要なVLANの存在 |
| 3 | L2 | SW1 | `show interfaces trunk` | Trunk・許可VLAN |
| 4 | L3 | R1 | `show running-config` | Subinterface・dot1Q設定 |
| 5 | L3 | R1 | `show ip interface brief` | Subinterfaceの動作状態 |

診断の流れは以下のようになります。

```text
L1
Interface状態
    ↓ 正常

L2
VLAN設定
    ↓ 正常

L2
Trunk設定
    ↓ 正常

L3
Subinterface・dot1Q設定
    ↓ 正常

L3
Subinterface状態
    ↓

すべて正常
または
障害を検出
```

## 4. 診断の仕組み

本システムはルールベースで診断を行います。

### 1. 診断手順を決定

選択された症状に応じて、以下の情報を取得します。

- 診断するレイヤー
- 確認対象機器
- 実行するCLIコマンド

### 2. コマンド出力を解析

利用者が入力したCLI出力から、診断に必要な情報を抽出します。

例えば以下の項目を確認します。

- VLAN 10、20、30が存在するか
- Trunkが設定されているか
- 必要なVLANがTrunkで許可されているか
- 必要なSubinterfaceが存在するか
- dot1Q設定が存在するか
- Subinterfaceがup/up状態か

### 3. 正常構成と比較

解析した結果を、あらかじめ定義した正常構成の期待値と比較します。

```text
CLIコマンド出力
        ↓
出力内容を解析
        ↓
正常構成の期待値と比較
        ↓
正常 / 異常を判定
```

### 4. 原因を特定

設定不足が検出された場合は、不足している設定内容を記録します。

例えば必要なSubinterfaceを一つずつ確認し、VLAN30用の設定だけが存在しない場合は、VLAN30を不足項目として取得します。

### 5. 復旧方法を提示

検出したエラーの種類と不足している設定情報をもとに、復旧手順とコマンド例を提示します。

## 5. ファイル構成

```text
network-troubleshooting-assistant/
│
├── app.py
├── diagnosis.py
├── analyzer.py
├── expected_config.py
├── troubleshooter.py
├── recovery.py
└── README.md
```

各ファイルの役割は以下のとおりです。

| ファイル | 役割 |
|---|---|
| `app.py` | UI表示と診断全体の進行を制御 |
| `diagnosis.py` | 症状ごとの診断手順、対象機器、確認コマンドを定義 |
| `analyzer.py` | 入力されたCLIコマンドの実行結果を解析 |
| `expected_config.py` | 正常なネットワーク構成の期待値を定義 |
| `troubleshooter.py` | 解析結果から正常・異常を判定し、障害原因を特定 |
| `recovery.py` | 障害に応じた復旧手順とコマンド例を生成 |

処理の関係は以下のとおりです。

```text
              diagnosis.py
                   ↓
app.py → analyzer.py ← expected_config.py
                   ↓
            troubleshooter.py
                   ↓
              recovery.py
```

- Python 3

```bash
pip install streamlit
```

## 7. 実行方法

プロジェクトのディレクトリへ移動します。

```bash
cd network-troubleshooting-assistant
```

以下のコマンドでアプリケーションを起動します。

```bash
streamlit run app.py
```

ブラウザ上にネットワーク診断アシスタントが表示されます。

---

## 8. 使用方法

1. Cisco Packet Tracerで対象ネットワークを開く
2. Streamlitアプリケーションを起動する
3. 発生している症状を選択する
4. 「診断開始」を押す
5. アプリケーションに表示された対象機器を開く
6. 指定されたCisco CLIコマンドを実行する
7. コマンドの実行結果をアプリケーションに貼り付ける
8. 「分析」を押す
9. 正常な場合は次の診断項目へ進む
10. 障害が検出された場合は原因と復旧方法を確認する

---

## 9. 診断例

### VLAN30用Subinterfaceが不足している場合

正常な構成では、VLAN 10、20、30に対応するSubinterfaceが必要です。

VLAN30用のSubinterfaceが存在しない場合、CLI出力を解析すると不足しているVLANとして30が検出されます。

```text
missing_subinterfaces = [30]
```

診断結果として、以下のような情報を表示します。

```text
障害を検出しました

障害内容
Subinterface設定エラー

原因
VLAN [30] 用のSubinterfaceがありません
```

さらに、VLAN30に対応した復旧コマンド例を提示します。

```text
interface <Routerの親Interface>.30
encapsulation dot1Q 30
ip address 192.168.30.254 255.255.255.0
```

※ Interface名やIPアドレスは実際のネットワーク構成に合わせる必要があります。

## 10. 開発中に発生した問題と改善

### 問題

当初は、コマンド出力に特定の文字列が含まれているかという単純な方法で正常・異常を判定していました。

そのため、設定に誤りや不足があっても、同じ文字列が別の場所に含まれていることで正常と判定される「すり抜け」が発生する場合がありました。

### 改善

正常なネットワーク構成を期待値として定義し、実際のコマンド出力と項目ごとに比較する方式へ変更しました。

また、必要に応じて正規表現を使用し、設定内容をより正確に抽出するようにしました。

さらに、不足している設定内容やVLAN番号も取得することで、障害の検出だけでなく、その後の原因表示や復旧コマンドの生成にも利用しています。

これにより、単純な文字列の存在確認から、正常構成との具体的な比較による判定へ改善しました。

## 11. 現在の制限

本システムは、Cisco Packet Tracer上に構築した特定のネットワーク構成を対象としたプロトタイプです。

現在は以下の制限があります。

- CLIコマンドの出力は手動で入力する必要がある
- 正常構成の期待値は事前に定義する必要がある
- 想定していない機器やコマンドの出力では正しく判定できない場合がある
- 複数の障害が存在する場合は、最初に検出した障害で診断を終了する
- PCのIPアドレスやSubnet Mask、Default Gatewayなど、一部の設定は自動診断の対象外
- ACLや動的ルーティングなどの複雑な障害には対応していない

## 12. 今後の改善案

今後の発展として、以下の機能が考えられます。

- SSHなどを利用したCLI出力の自動取得
- ネットワーク構成の自動認識
- 診断対象となる障害パターンの追加
- 複数障害の同時検出
- 入力されたCLI出力が正しい機器・コマンドのものかを検証する機能
- IPアドレスやDefault Gatewayの診断
- ACLやルーティングプロトコルの診断
- 異なるネットワーク構成に対応できる設定ファイルの導入

## 14. まとめ

本システムでは、Cisco機器のCLIコマンド出力を解析し、正常なネットワーク構成と比較することで、ネットワーク障害をL1からL3まで段階的に切り分けます。

正常な場合は次の診断項目へ進み、異常を検出した場合はその時点で原因を特定し、復旧手順とコマンド例を提示します。

これにより、ネットワーク障害の原因調査から復旧までを支援することを目的としています。
